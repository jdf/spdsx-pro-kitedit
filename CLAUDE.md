# spdsx-patchedit — orientation for Claude

Interactive kit designer for the Roland SPD-SX PRO. C++20, JUCE 8, CMake +
vcpkg. Public repo: github.com/jdf/spdsx-pro-kitedit (note: repo name differs
from this directory). Push with jdf's `jj push-main` alias.

## Conventions (STRICT — jdf cares)
- **Google C++ style throughout**: `.cc`/`.h` files, `#ifndef SPDSX_PATCHEDIT_SOURCE_<FILE>_H_` guards (never `#pragma once`), UpperCamelCase methods (trivial accessors like `name()`/`set_name()` may be snake_case), `kUpperCamelCase` enumerators/constants, `snake_case_` data members. JUCE overrides keep their inherited names.
- **JUCE string gotcha**: `juce::String` from a `char*` literal reads it byte-per-character — any non-ASCII UTF-8 (…, ·, —) becomes mojibake. Wrap such literals in `juce::String::fromUTF8(...)`, always.
- **jj (not git) for VCS**, colocated. Commit liberally in small semantic chunks (one logical change per commit, each builds and works; `jj commit <paths>` for per-file chunks). **Never push, and don't offer to** — jdf runs `jj push-main` himself when he wants. Co-author trailer: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
- Also honored in sibling repos: `github.com/jdf/specgram` (the spectrogram lib, dir `../specgram`) and `github.com/jdf/spdsx-py` (`../spdsx-py`, the protocol RE).

## Build / run
```
cmake --preset default && cmake --build --preset default
./build/spdsx-patchedit_artefacts/RelWithDebInfo/spdsx-patchedit.app/Contents/MacOS/spdsx-patchedit
```
Targets: the GUI app; `spdutil` (device CLI: `ping` / `info` / `dump` / `kits` / `kit <N>` / `padlink` / `selftest` — run it bare for the options); `spdsx_device` (JUCE-free static lib). Preset is RelWithDebInfo (breakpoints). VS Code: open `../hax.code-workspace` (multi-root); `--load <slot> <wav>` pre-fills slots for testing.

## Architecture (source/)
- `kit_model.{h}` — the **active kit**: name + 9 `Pad`s. A layer holds a `LayerSample` — **dual identity**: a local audio file OR a device pool wave (`device_index` > 0; metadata-only until sample transfer is cracked, so it displays but can't play; at sync time local files get uploaded and become indices too). Each pad also carries a `PadParams` (layer mode, fade point/end — end constrained >= point, defaults 80/127 matching the device factory — dynamics on/off, dynamics curve, fixed velocity — the strike level every hit plays at when dynamics is off — trigger reserve, which is read/write only, not emulated, and the HI-HAT closed-pedal trio hi_hat_volume/hi_hat_fade_in/hi_hat_decay, of which only volume is emulated — it scales the closed layer in TriggerPad; fade in/decay await engine envelopes). `PadParams` mutates as a unit: one setter, one action, one listener event. Observable (`KitModel::Listener`). UI/engine/document all react to it.
- `device_model.h` — the **whole-device mirror**: 200 `KitData` (name + pads) as plain data, plus which kit is active, plus the **sample pool** (`sample_pool()`/`FindSample(index)` — the device's wave directory as `device::SampleRecord`s, metadata only). Not observable; the active kit is edited through KitModel and stashed back by the document.
- `layers.{h,cc}` — JUCE-free pure logic emulating the device's 8 layer types (MIX/FADE1/FADE2/XFADE/SWITCH/SW(MONO)/ALTERNATE/HI-HAT): `ComputeLayerWeights(...)` -> per-layer selection weights (0..1) + choke flag, deliberately excluding loudness; `DynamicsGain(curve, velocity)` -> loudness through the Dynamics Curve (LINEAR/LOUD1/LOUD2/LOUD3 — power-curve approximations, real device shapes unpublished). Selection always follows real strike velocity; loudness is weights × DynamicsGain (with Dynamics off, × the linear gain at the pad's fixed velocity). Mode/curve names are shared by the UI and the JSON persistence.
- `actions.h` — `SetSampleAction`, `SetPadParamsAction` (undoable). `main_component` wraps edits in `undo_` transactions (Load, MoveSample, MovePad, pad-param changes).
- `device_document.{h,cc}` — `DeviceDocument : FileBasedDocument`. The document is a FOLDER `*.spdsx` (a macOS package: Info.plist via CMake `PLIST_TO_MERGE` declares `com.jdf.spdsx-device` conforming to `com.apple.package` with `LSTypeIsPackage`) holding `device.json` (`DeviceFormat` v1: format, currentKit, 200 kits; pads in the same JSON shape legacy .kit v4 used; absolute sample paths, pool indices come with device sync; a future `samples/` cache is preserved on save). `SwitchKit` stashes the active kit, loads the next, and preserves the unsaved-changes flag (a kit switch is view state, not an edit). Undo histories are PER-KIT (`MainComponent::undo()` returns the current kit's lazily-created UndoManager) and survive switches — safe because actions only ever enter the manager of the kit that was live, so they replay against the right kit; document-level events (open/new/import) clear all histories via `on_history_reset`. `ImportKitFile` reads legacy single-kit `.kit` v1-v4 into the active kit (File > Import Kit…). ALL device opening goes through `OpenDevice(file-or-folder)` — JUCE's `FileBasedDocument::loadFrom` insists on `existsAsFile()`, which a folder document never satisfies, so File > Open uses its own FileChooser and the app auto-opens the last document on startup (`MainComponent::OpenLastDocument`, skipped when `--load` args or `SPDSX_SELFTEST` are given). **NO EXPLICIT SAVE — everything autosaves** (jdf decision 2026-07-12): every mutation calls `MarkEdited()` (stamps time + `changed()`), the 30Hz timer flushes via `DeviceDocument::Autosave()` after a 1s quiet period, quit flushes synchronously. The document is always file-backed: startup falls back to opening/creating `~/Documents/SPD-SX PRO.spdsx`; File > New asks for a location up front (`CreateNew` saves immediately); no Save menu/cmd-S, no dirty dot/"Edited" marker, no save prompts; Save As remains as "move the document, autosaves follow". `OpenDevice` also accepts the `device.json` path directly (fallback when package registration hasn't kicked in — run `lsregister -f <app>` after building to a fresh location). NOTE: FileBasedDocument's sync save/load are compiled out in JUCE 8 (no modal loops) — selftests go through temporary hooks calling the protected saveDocument/loadDocument.
- `kit_chooser.{h,cc}` — the header's unified kit control: prev/next arrows around a chip with kit number + name; clicking the chip drops the 200-kit menu, the pencil starts in-place rename. Pure view: callbacks out (`on_select`/`on_rename`), `SetCurrent` in, menu names via the `kit_name` provider (live model name for the active kit).
- `main_component.{h,cc}` — the grid, the KitChooser wiring (switching kits via DeviceDocument::SwitchKit), undo, File/Edit/View menus (via ApplicationCommandManager; `main.cc` must call `setApplicationCommandManagerToWatch` or menu enablement goes stale), MIDI in (ch 10 notes 60-68 -> velocity-aware `TriggerPad` through the layer mode; CC4 = hi-hat pedal), click/space anywhere in a pad = whole-pad hit through the layer mode with cursor HEIGHT in the pad as velocity (bottom=1, top=127; slot clicks route via on_click, pad-surface clicks via mouseDown), hits from any source flash the pad in a velocity colour (blue→amber→red, `VelocityColour` in sample_slot) and tint each sounding layer with its adjusted velocity while it plays, keys 1-9 trigger pads at the header VEL knob's velocity (a compact rotary with a click-to-type value box, dial tinted the velocity colour; persisted as `uiVelocity`), a green header transfer button ("↓ N samples") appears when the active kit references device waves not yet cached and downloads them on click (`UpdateTransferButton` from `UncachedDeviceWaveCount`), H key held = hi-hat pedal down (press = foot-close "chick": cuts the open layer, sounds the closed one; release silent; key-up detected via `keyStateChanged` + `isKeyCurrentlyDown` polling — `keyPressed` never reports releases), per-pad mode combo + fade sliders + "⋯" button in each pad header (opens the `pad_settings` CallOutBox), drag (move/copy one slot; ⌘ = whole pad; ⌥ = duplicate). Slot-level transport/space/click stays full-volume, ignoring layer mode.
- `pad_settings.{h,cc}` — the pad "⋯" panel (shown in a `CallOutBox` anchored to the button): checkboxes for Dynamics and Trigger Reserve, a ComboBox for the curve, rotary knobs with click-to-type value boxes for Fixed Velocity and — on HI-HAT pads only — the "Closed Pedal" section (Volume/Fade In/Decay; the panel grows and the CallOutBox follows). Curve enabled when dynamics is on, fixed velocity when it's off (jdf's UI rule: real controls, not menu items — knobs for scalars with direct numeric entry, checkboxes for booleans). Pure view: `SetParams` in, `on_change` out (only the fields it owns get merged back, so header edits to mode/fades can't be clobbered); `MainComponent::PadParamsChanged` refreshes an open panel so undo/redo shows through live. The fade point/end pair stays in the pad header (LinearBars); the end slider's minimum tracks the fade point, so the >= constraint holds structurally. JUCE gotcha (bitten 2026-07-12): a LinearBar's default `trackColourId` fill is indistinguishable from our dark background — with NoTextBox that renders as a dead-looking empty box. The fade bars (and the header VEL bar) are therefore tinted per-value with `VelocityColour` (the pad-flash blue→amber→red language) at 0.5 alpha, and the fade bars get `setPopupDisplayEnabled` for a value bubble while dragging.
- `sample_slot.{h,cc}` — a slot: spectrogram + info bar + transport buttons; drag source + FileDragAndDropTarget + DragAndDropTarget; hover highlight driven by parent (`on_drag_target`/`SetDragTarget`).
- `device_samples.{h,cc}` — the left panel's "Device" tab (the panel is now a TabbedComponent: "Files" = sample_browser, "Device" = this): the wave pool as a ListBox (name + category per row) under a live case-insensitive substring filter box (matches wavename or filename); rows drag onto slots with description `spdsx-devsample:<index>` (`kDeviceSampleDragPrefix` in audio_files.h). Selection honors View > Autoplay exactly like the file browser and fires `on_preview` — a documented no-op until the sample cache delivers audio. Pool fills via **File > Load Device State** (the whole-document read; see Open items). **File > Download Kit Samples** (`MainComponent::DownloadKitSamples`) then pulls the ACTIVE kit's device waves into a local cache: on a worker thread `FindDevicePort` + per wave `ReadRemoteWave` (the f0 41 7a file read) + `RfwvToWav`, written to `<bundle>/samples/NNNNN.wav` (`DeviceDocument::WaveCacheFile`); each cached wave refreshes its slots via `OnWaveCached`. A slot with a cached device wave loads and renders exactly like a local file (`LoadAudioIntoSlot`, shared with the file path); an uncached one shows name + duration + "on device" (`SetDeviceSample`). Device-tab autoplay previews a wave once cached. Both preloads and user imports are cacheable/playable (the device serves preload `.SMP` files over the file protocol even though the official app's UI won't export them); the Device tab just dims preloads as a factory/user cue.
- `sample_browser.{h,cc}` — left panel FileTree; drags onto slots; autoplay (View menu, persisted). Root folder AND tree state (open folders/scroll/selection, via TreeView openness XML in settings) persist across runs; restore re-applies on a timer until the async directory scans catch up (one shot can't reopen nested folders).
- `audio.{h,cc}` — AudioEngine (per-slot transports + a preview channel), miniaudio-free (pure JUCE).
- `spectro.{h,cc}` — renders spectrogram thumbnails via specgram (consumed through the vcpkg overlay port in `ports/specgram`, pinned to a `github.com/jdf/specgram` sha; bump REF there after pushing specgram).
- `device/` — SPD-SX PRO serial protocol (JUCE-free): `protocol` (DT1 message building, addresses, nibble codec, bulk-read requests + `SplitBulkImage`), `serial_port` (termios + macOS IOSSIOSPEED), `spdsx_device` (frame wrap/unwrap + ops), `kit_image` (`CleanBulkImage` strips block framing; `ParseKits` reads the kit-record array), `sample_image` (`ParseSampleDir` reads the bank 0x20 wave-pool directory; `RemoteWavePath`, `ParseRfwvHeader`, `RfwvToWav`). `spdsx_device` ops now include: `DumpBank` (bulk read), `ReadRemoteWave` (the `f0 41 7a` channel-0x06 file read of a wave's `.SMP`, LIVE-verified), `WriteRemoteFile` (file write, LIVE-verified byte-exact — the upload write path), `DeleteWave` (LIVE-verified), and `Commit` (the `6a 03 21`/`22` flash handshake). CLI: `spdutil readwave <N> [--out f.wav|f.smp]`, `deletewave <N>`, `sendwave <N> --from f.smp` (write+readback test), plus `dump`/`kits`/`kit`/`samples`/`info`/`ping`/`padlink`. The GUI app links this lib too. See `spdutil selftest`.
- **Kit record layout** (RE'd 2026-07-12, ground-truth verified: ZZZ kit = #129 and edited pad values read back exactly): bank 0x10, clean (header-stripped) image, array base **0x262c**, stride **3528** bytes, **200** records; name = first **16** bytes (space-padded ASCII). Per-pad param table at **rec+0x284**, 9 pads × **28-byte** blocks; within a block the byte offset = the DT1 param index: +0x00 layer mode (MIX=0…HI-HAT=7), +0x01 fade point, +0x02 fade end, +0x03 dynamics on/off, +0x04 dynamics curve (LINEAR=0…LOUD3=3), +0x05 **fixed velocity** (used when dynamics off), +0x13 trigger reserve. HYPOTHESIS from factory-image inspection (2026-07-12, unverified — confirm by live diff): +0x06/+0x07/+0x08 = hi-hat fade-in/volume/decay (USER KIT holds 0/80/25 there, and they vary on factory HI-HAT pads); the GUI's closed-pedal defaults were seeded from that reading. `device/kit_image` parses names + these into `KitRecord`. **Per-layer table at rec+0x49a** (mapped 2026-07-12 offline against the ZZZ-kit ground truth): 18 layers (9 pads × top/bottom) × 60-byte blocks, each starting with the layer's WAVE (pool index, u16 LE, 0 = none). EQ/effects/kit-level settings still unmapped (same diff method). `spdutil kits [--from <dump>]` lists names; `spdutil kit <N> [--from <dump>]` shows a kit's pad params.
- **Pad-param WRITE map** (captured live via `re/frida-scripts/paramlog.js`, 2026-07-12): DT1 `06 00 <0x1F+pad> <param>` with the SAME param index as storage, preceded by an object focus `28 00 00 00 <pad-1>`. Values match the enums 1:1. Not yet ported to `spdsx_device` write ops (the "my copy wins" side of sync).
- **Flash-commit handshake — the WRITE button DOES send serial commands** (corrected 2026-07-12; an earlier note wrongly said "nothing sent" because an `isPoll` filter suppressed all `6a 03` frames, eating exactly these). On WRITE the app sends, on ch 0x09 control family: `f0 41 6a 03 21 ...` (begin commit) → device acks `f0 41 6a 7a ...`; then it polls `f0 41 6a 03 22 ...` every ~250ms, each answered `f0 41 6a 02 ... 22 40 00 00 00 04 <status LE32> f7` with status 0 while busy, flipping to 1 when the flash write completes (~1.8s). So durable sync WRITE = the mapped DT1 param writes, THEN `6a 03 21` + poll `6a 03 22` until status==1. (Bonus from the same capture: `6a 03 cat=17` returns the firmware version ASCII, e.g. "2.00"/"0094" — cracks `spdutil info`. `7a` replies are generic ACKs, seen on both 6a and 6c families.) Full capture: `../re-cache/captures/writecmd-commit-handshake.log`.

## Device reverse-engineering (the big ongoing thread)
NOT MIDI — it's Roland DT1 SysEx wrapped in a proprietary transport frame over USB CDC-ACM serial (230400 baud). See `[[project-spdsx-patchedit]]` memory for the full protocol map. Cracked so far: kit select, object focus, pad-link, kit name (read+write), pad wave assignment (read+write, nibble-encoded), per-pad layer params (read+write — the kit-record layout above), the full bulk-read handshake, the flash-commit (WRITE) handshake, and the firmware-version query. Device state = ~8MB bulk block transfer on the `41 6c` family, banked; the cached image holds TWO passes of every bank (the app's load runs twice), so per-pass sizes are half the totals. 0x10 = kit/settings (~0.7MB/pass, 3528-byte-per-kit records); 0x20 = the **sample-pool DIRECTORY, not audio** (~3.3MB/pass: 20001 × 164-byte records — wavename/filename/frames/category + an opaque u64 pointer into device flash — then category + user-tag name tables; layout in `device/sample_image.h`). **No wave audio crosses in the state dump at all**, and the official app bundle ships none either: sample download/upload is a separate, still-uncaptured protocol. Method to map a parameter: dump image, change ONE thing in the official app, dump again, diff. Capture + RE tooling lives in **`re/`** (frida capture scripts, `parse_capture.py`, and the original `spdsx.py`/`link_all_kits.py` Python reference this C++ was ported from; see `re/README.md`). frida at `~/frida-env/bin/frida`, SIP disabled; keep attached via `tail -f /dev/null | frida -f <app> -l <script>`. An offline ~8MB device image is cached at **`../re-cache/device-image.bin`** with a memo — the source for all documented storage offsets. jdf drives the GUI + hardware for captures. (Also mirrored at github.com/jdf/spdsx-py.)

**Live transport VERIFIED** (2026-07-11, re-verified after the spdutil rename): `spdutil ping` opened the port and pinged the real device, which replied byte-identically to the frida capture (`f0 41 6a 02 ... 16 40 00 00 00 04 01 00 00 00 f7`). So the whole C++ transport stack (serial open + baud ioctl, frame wrap/unwrap, round-trip) works against hardware. The `/dev/cu.usbmodem<N>` node number changes every replug — there is no stable path; with no `--port`, spdutil scans `/dev/cu.usbmodem*` and pings each node until the device answers. Close the official app first (one program per port). Bulk reads are exercised live too (`spdutil dump` / `kits` / `kit <N>` all ran against hardware 2026-07-12); the write paths (SetKitName/SetPadWave) are still unexercised live.

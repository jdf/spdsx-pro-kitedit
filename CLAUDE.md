# spdsx-patchedit — orientation for Claude

Interactive kit designer for the Roland SPD-SX PRO. C++20, JUCE 8, CMake +
vcpkg. Public repo: github.com/jdf/spdsx-pro-kitedit (note: repo name differs
from this directory). Push with jdf's `jj push-main` alias.

## Conventions (STRICT — jdf cares)
- **Google C++ style throughout**: `.cc`/`.h` files, `#ifndef SPDSX_PATCHEDIT_SOURCE_<FILE>_H_` guards (never `#pragma once`), UpperCamelCase methods (trivial accessors like `name()`/`set_name()` may be snake_case), `kUpperCamelCase` enumerators/constants, `snake_case_` data members. JUCE overrides keep their inherited names.
- **The tree is clang-format clean — just run it** (`clang-format -i <files>`; it is also VS Code's formatter for C++ via clangd). `.clang-format` is authoritative and agrees with the code: braces attach to the opening line (`void Foo() {`), a file's own header includes first, then std, then other libraries, then project headers. This was NOT true before 2026-07-14 — the config and the code disagreed and formatting a file rewrote it wholesale — so ignore any older note telling you to hand-match the surrounding style.
- **JUCE string gotcha**: `juce::String` from a `char*` literal reads it byte-per-character — any non-ASCII UTF-8 (…, ·, —) becomes mojibake. Wrap such literals in `juce::String::fromUTF8(...)`, always.
- **jj (not git) for VCS**, colocated. Commit liberally in small semantic chunks (one logical change per commit, each builds and works; `jj commit <paths>` for per-file chunks). **Never push, and don't offer to** — jdf runs `jj push-main` himself when he wants. Co-author trailer: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
- Also honored in sibling repos: `github.com/jdf/specgram` (the spectrogram lib, dir `../specgram`) and `github.com/jdf/spdsx-py` (`../spdsx-py`, the protocol RE).

## Build / run
```
cmake --preset default && cmake --build --preset default
./build/spdsx-patchedit_artefacts/RelWithDebInfo/spdsx-patchedit.app/Contents/MacOS/spdsx-patchedit
```
`./install-app.sh` builds and installs `/Applications/spdsx-patchedit.app` — a normal double-clickable app with its own icon (`assets/icon.svg` → `icon.png` → JUCE `ICON_BIG`; regen recipe in `assets/README.md`) and the `.spdsx` extension registered, so double-clicking a document opens it (single-instance; `App::anotherInstanceStarted` → `MainComponent::OpenDocument`).
Targets: the GUI app; `spdutil` (device CLI: read `ping`/`info`/`dump`/`kits`/`kit <N>`/`samples`/`readwave`; write `assign`/`setname`/`setparams`/`sendwave` (upload+register)/`deletewave`/`padlink` (write ops take `--commit`) — run it bare for the options); `spdsx_device` (JUCE-free static lib). Preset is RelWithDebInfo (breakpoints). VS Code: open `../hax.code-workspace` (multi-root); `--load <slot> <wav>` pre-fills slots for testing.

## Architecture (source/)
- `kit_model.{h}` — the **active kit**: name + 9 `Pad`s. A layer holds a `LayerSample` — **dual identity**: a local audio file OR a device pool wave (`device_index` > 0; metadata-only until sample transfer is cracked, so it displays but can't play; at sync time local files get uploaded and become indices too). Each pad also carries a `PadParams` (layer mode, fade point/end — end constrained >= point, defaults 80/127 matching the device factory — dynamics on/off, dynamics curve, fixed velocity — the strike level every hit plays at when dynamics is off — trigger reserve, which is read/write only, not emulated, and the HI-HAT closed-pedal trio hi_hat_volume/hi_hat_fade_in/hi_hat_decay, of which only volume is emulated — it scales the closed layer in TriggerPad; fade in/decay await engine envelopes). `PadParams` mutates as a unit: one setter, one action, one listener event. Observable (`KitModel::Listener`). UI/engine/document all react to it.
- `device_model.h` — the **whole-device mirror**: 200 `KitData` (name + pads) as plain data, plus which kit is active, plus the **sample pool** (`sample_pool()`/`FindSample(index)` — the device's wave directory as `device::SampleRecord`s, metadata only). Not observable; the active kit is edited through KitModel and stashed back by the document.
- `layers.{h,cc}` — JUCE-free pure logic emulating the device's 8 layer types (MIX/FADE1/FADE2/XFADE/SWITCH/SW(MONO)/ALTERNATE/HI-HAT): `ComputeLayerWeights(...)` -> per-layer selection weights (0..1) + choke flag, deliberately excluding loudness; `DynamicsGain(curve, velocity)` -> loudness through the Dynamics Curve (LINEAR/LOUD1/LOUD2/LOUD3 — power-curve approximations, real device shapes unpublished). Selection always follows real strike velocity; loudness is weights × DynamicsGain (with Dynamics off, × the linear gain at the pad's fixed velocity). Mode/curve names are shared by the UI and the JSON persistence.
- `actions.h` — `SetSampleAction`, `SetPadParamsAction` (undoable). `main_component` wraps edits in `undo_` transactions (Load, MoveSample, MovePad, pad-param changes).
- `device_db.{h,cc}` — the persistence store: a single `*.spdsx` **SQLite** file (via vcpkg `unofficial-sqlite3`). Schema: `meta` (schema_version/app/created), `kits`+`pads` each carrying a `snapshot` column (`current` = the working doc, `base` = the last-synced state for 3-way sync), and `samples` (pool directory metadata + cached audio as a BLOB). `DeviceDb` reads/writes a `DeviceModel` (`ReadKits`/`WriteKits` per snapshot, `WritePool`, `HasAudio`/`GetAudio`/`PutAudio`, `CaptureBase`). Covered by `device_db_test.cc`.
- `device_sync.{h,cc}` — the **three-way sync engine**, pure logic (no I/O): `MergePad` does a FIELD-wise current/base/theirs merge (a field changed on one side takes that side; changed apart = conflict, resolved per pad by `SyncResolution` mine/theirs/skip — skip leaves the pad AND its base untouched so it re-flags next sync); `FindKitConflicts` renders dialog-ready descriptions with both values; `PlanKitSync` yields a `KitSyncPlan` (new_current/new_base + which name/params/waves need device writes — a theirs-wins conflict needs NO write, it pulls); `PlanUploads`/`SubstituteUploads` assign free pool indices (`NextFreeSampleIndex`) to distinct local files in written layers and swap them to `DeviceWave` in both snapshots; `BuildKitWrite` lowers a plan to wire-level `KitWrite`s; `ExecutePush(dev, uploads, kits, on_uploaded, pace, commit_timeout)` drives UploadWave (each durable, fires on_uploaded) then name/wave/param writes then ONE `Commit`. Also home of `KitDataFromDevice` (shared with Load Device State) and `DeviceParamsFromPad`. Fully covered in `device_sync_test.cc`, including pushes against the fake port and an end-to-end plan→wire test.
- `sample_upload.{h,cc}` — `SmpFromAudioFile`: decode any JUCE-readable file, resample to 48 kHz (Lagrange, the only rate the device plays), first two channels, 16-bit, wrap via `PcmToRfwv` (checksum included). Refuses non-audio/empty/absurdly-long input with a message.
- `sync_dialog.{h,cc}` — `SyncConflictPanel`: the conflict-resolution dialog (async modal `DialogWindow`), one row per conflicted pad/kit-name with a ComboBox (My copy wins / Device wins / Do nothing, defaulting to mine since the user pressed Save); Sync/Cancel buttons; any other close path (escape, close button) reports cancel via the destructor. Pure view.
- `device_document.{h,cc}` — `DeviceDocument : FileBasedDocument`, bridging the in-memory model to `DeviceDb`. The document is a single SQLite FILE `*.spdsx` (no more folder-package/UTI/PLIST — `existsAsFile()` just works). `loadDocument`/`saveDocument`/`Autosave`/`CreateNew`/`OpenDevice` all go through `db_`; `SwitchKit` stashes the active kit, loads the next, preserves the unsaved flag (view state, not an edit). Undo histories are PER-KIT (`MainComponent::undo()` returns the current kit's lazily-created UndoManager) and survive switches; document-level events (open/new/import) clear all via `on_history_reset`, and `on_model_reload` brackets `LoadActiveKitIntoModel` so a load's change-listener storm isn't mistaken for a user edit. `ImportKitFile` still reads legacy single-kit `.kit` v1-v4 into the active kit (File > Import Kit…). Cached wave audio lives as DB blobs: `StoreWaveAudio` (from a download), `HasCachedAudio`, and `CachedWaveFile` (extract-on-demand to a temp file the engine can load). `ReplaceWithDeviceState` (Load Device State) writes kits+pool AND `CaptureBase()` (device == current == base). **NO EXPLICIT SAVE — everything autosaves** (`Autosave()` = `db_->WriteKits` after a 1s quiet period; quit flushes). Startup opens the last file or creates `~/Documents/SPD-SX PRO.spdsx`; `CreateNew` blows away anything (incl. a legacy folder) at that path. All prior-document JSON read paths were dropped (jdf approved; old folder docs won't open). NOTE: FileBasedDocument's sync save/load are compiled out in JUCE 8, so the app drives loadDocument/saveDocument through OpenDevice/CreateNew/Autosave; `device_document_test.cc` reaches the protected ones with a `using`-subclass. Covered by that suite.
- `kit_chooser.{h,cc}` — the header's unified kit control: prev/next arrows around a chip with kit number + name; clicking the chip drops the 200-kit menu, the pencil starts in-place rename. Pure view: callbacks out (`on_select`/`on_rename`), `SetCurrent` in, menu names via the `kit_name` provider (live model name for the active kit).
- `main_component.{h,cc}` — the grid, the KitChooser wiring (switching kits via DeviceDocument::SwitchKit), undo, File/Edit/View menus (via ApplicationCommandManager; `main.cc` must call `setApplicationCommandManagerToWatch` or menu enablement goes stale), MIDI in (ch 10 notes 60-68 -> velocity-aware `TriggerPad` through the layer mode; CC4 = hi-hat pedal), click/space anywhere in a pad = whole-pad hit through the layer mode with cursor HEIGHT in the pad as velocity (bottom=1, top=127; slot clicks route via on_click, pad-surface clicks via mouseDown), hits from any source flash the pad in a velocity colour (blue→amber→red, `VelocityColour` in sample_slot) and tint each sounding layer with its adjusted velocity while it plays, keys 1-9 trigger pads at the header VEL knob's velocity (a compact rotary with a click-to-type value box, dial tinted the velocity colour; persisted as `uiVelocity`), a green header transfer button ("↓ N samples") appears when the active kit references device waves not yet cached and downloads them on click (`UpdateTransferButton` from `UncachedDeviceWaveCount`), H key held = hi-hat pedal down (press = foot-close "chick": cuts the open layer, sounds the closed one; release silent; key-up detected via `keyStateChanged` + `isKeyCurrentlyDown` polling — `keyPressed` never reports releases), per-pad mode combo + fade sliders + "⋯" button in each pad header (opens the `pad_settings` CallOutBox), drag (move/copy one slot; ⌘ = whole pad; ⌥ = duplicate). Slot-level transport/space/click stays full-volume, ignoring layer mode.
- `pad_settings.{h,cc}` — the pad "⋯" panel (shown in a `CallOutBox` anchored to the button): checkboxes for Dynamics and Trigger Reserve, a ComboBox for the curve, rotary knobs with click-to-type value boxes for Fixed Velocity and — on HI-HAT pads only — the "Closed Pedal" section (Volume/Fade In/Decay; the panel grows and the CallOutBox follows). Curve enabled when dynamics is on, fixed velocity when it's off (jdf's UI rule: real controls, not menu items — knobs for scalars with direct numeric entry, checkboxes for booleans). Pure view: `SetParams` in, `on_change` out (only the fields it owns get merged back, so header edits to mode/fades can't be clobbered); `MainComponent::PadParamsChanged` refreshes an open panel so undo/redo shows through live. The fade point/end pair stays in the pad header (LinearBars); the end slider's minimum tracks the fade point, so the >= constraint holds structurally. JUCE gotcha (bitten 2026-07-12): a LinearBar's default `trackColourId` fill is indistinguishable from our dark background — with NoTextBox that renders as a dead-looking empty box. The fade bars (and the header VEL bar) are therefore tinted per-value with `VelocityColour` (the pad-flash blue→amber→red language) at 0.5 alpha, and the fade bars get `setPopupDisplayEnabled` for a value bubble while dragging.
- `sample_slot.{h,cc}` — a slot: spectrogram + info bar + transport buttons; drag source + FileDragAndDropTarget + DragAndDropTarget; hover highlight driven by parent (`on_drag_target`/`SetDragTarget`).
- `device_samples.{h,cc}` — the left panel's "Device" tab (the panel is now a TabbedComponent: "Files" = sample_browser, "Device" = this): the wave pool as a ListBox (name + category per row) under a live case-insensitive substring filter box (matches wavename or filename); rows drag onto slots with description `spdsx-devsample:<index>` (`kDeviceSampleDragPrefix` in audio_files.h). Selection honors View > Autoplay exactly like the file browser and fires `on_preview` — a documented no-op until the sample cache delivers audio. Pool fills via **File > Load Device State** (the whole-document read; see Open items). **File > Download Kit Samples** (`MainComponent::DownloadKitSamples`) then pulls the ACTIVE kit's device waves into a local cache: on a worker thread `FindDevicePort` + per wave `ReadRemoteWave` (the f0 41 7a file read) + `RfwvToWav`, then handed back to the message thread (`OnWaveDownloaded`) which stores it as a DB blob (`DeviceDocument::StoreWaveAudio`); each cached wave refreshes its slots via `OnWaveCached`. A slot with a cached device wave loads and renders exactly like a local file (`LoadAudioIntoSlot`, shared with the file path); an uncached one shows name + duration + "on device" (`SetDeviceSample`). Device-tab autoplay previews a wave once cached. Both preloads and user imports are cacheable/playable (the device serves preload `.SMP` files over the file protocol even though the official app's UI won't export them); the Device tab just dims preloads as a factory/user cue.
- `sample_browser.{h,cc}` — left panel FileTree; drags onto slots; autoplay (View menu, persisted). Root folder AND tree state (open folders/scroll/selection, via TreeView openness XML in settings) persist across runs; restore re-applies on a timer until the async directory scans catch up (one shot can't reopen nested folders).
- `audio.{h,cc}` — AudioEngine (per-slot transports + a preview channel), miniaudio-free (pure JUCE).
- `spectro.{h,cc}` — renders spectrogram thumbnails via specgram (consumed through the vcpkg overlay port in `ports/specgram`, pinned to a `github.com/jdf/specgram` sha; bump REF there after pushing specgram).
- `device/` — SPD-SX PRO serial protocol (JUCE-free): `protocol` (DT1 message building, addresses, nibble codec, bulk-read requests + `SplitBulkImage`), `serial_port.h` (the `SerialPort` byte channel + `PortBackend` = what nodes might be the device and how to open one — both interfaces, NO platform named; `serial_port_macos.cc` is the one implementation, termios + IOSSIOSPEED + the /dev/cu.usbmodem* glob, all file-local behind `PlatformPorts()`, and CMake compiles it only on APPLE, failing at configure elsewhere), `spdsx_device` (frame wrap/unwrap + ops; **borrows** a `SerialPort*`, which is what lets a fake drive every op), `kit_image` (`CleanBulkImage` strips block framing; `ParseKits` reads the kit-record array), `sample_image` (`ParseSampleDir` reads the bank 0x20 wave-pool directory; `RemoteWavePath`, `ParseRfwvHeader`, `RfwvToWav`). `spdsx_device` ops now include: `DumpBank` (bulk read), `ReadRemoteWave` (the `f0 41 7a` channel-0x06 file read of a wave's `.SMP`, LIVE-verified), `WriteRemoteFile` (file write, LIVE-verified byte-exact — the upload write path), `DeleteWave` (LIVE-verified), and `Commit` (the `6a 03 21`/`22` flash handshake). Also `PcmToRfwv` (build a playable `.SMP` incl. the MD5 header checksum), `SetKitName`/`SetPadWave`/`SetPadLayerParams` (all kit-absolute, live-verified), and `UploadWave` (write+register). CLI write ops (all `--commit`-able): `assign <K> --sample N --pad P.S`, `setname <K> --name ...`, `setparams <K> --pad P --params ...`, `sendwave <N> --from f.smp [--name X.wav]` (upload+register), `deletewave <N>`; reads `readwave`/`dump`/`kits`/`kit`/`samples`/`info`/`ping`. The GUI app links this lib too. The capture-derived byte-exact checks live in `source/device/*_test.cc` (they used to be `spdutil selftest`, which ran only when someone typed it; ctest and presubmit run them now).
- **Kit record layout** (RE'd 2026-07-12, ground-truth verified: ZZZ kit = #129 and edited pad values read back exactly): bank 0x10, clean (header-stripped) image, array base **0x262c**, stride **3528** bytes, **200** records; name = first **16** bytes (space-padded ASCII). Per-pad param table at **rec+0x284**, 9 pads × **28-byte** blocks; within a block the byte offset = the DT1 param index: +0x00 layer mode (MIX=0…HI-HAT=7), +0x01 fade point, +0x02 fade end, +0x03 dynamics on/off, +0x04 dynamics curve (LINEAR=0…LOUD3=3), +0x05 **fixed velocity** (used when dynamics off), +0x07/+0x08/+0x09 hi-hat closed-pedal **volume/fade-in/decay** (mapped live 2026-07-13 by capturing the app's writes on a hi-hat pad — this supersedes and corrects the earlier +0x06/07/08 fade-in/volume/decay guess, which was off by one and mislabeled; USER KIT stores volume 80 @0x07, fade-in 0 @0x08, decay 25 @0x09), +0x13 trigger reserve. `device/kit_image` parses names + these into `KitRecord`; `SpdsxDevice::SetPadLayerParams` writes them. **Per-layer table at rec+0x49a** (mapped 2026-07-12 offline against the ZZZ-kit ground truth): 18 layers (9 pads × top/bottom) × 60-byte blocks, each starting with the layer's WAVE (pool index, u16 LE, 0 = none). EQ/effects/kit-level settings still unmapped (same diff method). `spdutil kits [--from <dump>]` lists names; `spdutil kit <N> [--from <dump>]` shows a kit's pad params.
- **Pad-param WRITE map** (captured live via `re/frida-scripts/paramlog.js`, 2026-07-12): DT1 `<kit-prefix> <0x1F+pad> <param>` with the SAME param index as storage. **The first two address bytes are the kit** (`PadLinkPrefix(kit)` = `512+2*(kit-1)` split 7-bit) — captures were on kit 129 so they show `06 00`, which is exactly kit 129's prefix; the writes are kit-absolute, NOT current-kit-relative (corrected 2026-07-13 after every write initially hit kit 129). Values match the enums 1:1. Ported: `SetPadLayerParams`/`SetPadWave`/`SetKitName` build the kit prefix, no SelectKit needed.
- **Flash-commit handshake — the WRITE button DOES send serial commands** (corrected 2026-07-12; an earlier note wrongly said "nothing sent" because an `isPoll` filter suppressed all `6a 03` frames, eating exactly these). On WRITE the app sends, on ch 0x09 control family: `f0 41 6a 03 21 ...` (begin commit) → device acks `f0 41 6a 7a ...`; then it polls `f0 41 6a 03 22 ...` every ~250ms, each answered `f0 41 6a 02 ... 22 40 00 00 00 04 <status LE32> f7` with status 0 while busy, flipping to 1 when the flash write completes (~1.8s). So durable sync WRITE = the mapped DT1 param writes, THEN `6a 03 21` + poll `6a 03 22` until status==1. (Bonus from the same capture: `6a 03 cat=17` returns the firmware version ASCII, e.g. "2.00"/"0094" — cracks `spdutil info`. `7a` replies are generic ACKs, seen on both 6a and 6c families.) Full capture: `../re-cache/captures/writecmd-commit-handshake.log`.

## Unit tests & coverage (`testing/` + `source/*_test.cc`)
gtest suite (vcpkg `gtest`). **Convention: for every `source/foo.h` there is a `source/foo_test.cc`** holding that header's `TEST()`s, next to the code it covers. Each suite is its own TU, listed in the `spdsx_tests` target in CMakeLists.txt (add a line per new suite); it includes its header under test first, and keeps test-local helpers/fixtures in an anonymous namespace inside `namespace spdsx`. Suite names are global to the binary — keep them distinct. Test-only helpers live in `testing/` (`juce_test_environment.{h,cc}` = a gtest global env bringing up `juce::ScopedJuceInitialiser_GUI` for the session, registered so no suite has to; `temp_dir.h` = RAII temp dir for DeviceDb/DeviceDocument; `fake_serial_port.h` = a SerialPort that answers with queued replies and records writes; `fake_port_backend.h` = a PortBackend of fake nodes, some refusing to open). The `spdsx_tests` target is a JUCE console app compiling every production TU (minus the `main.cc`/`spdutil_main.cc` entry points) so most headers are testable; built by default. Run: `cmake --build --preset default --target spdsx_tests && ctest --test-dir build` (`gtest_discover_tests` registers each TEST as its own ctest entry, so `-R` filters per test; the binary also takes `--gtest_filter=` etc. directly). VS Code: the Testing panel lists each TEST with gutter run/debug via the **TestMate C++** extension (settings + CodeLLDB template in `.vscode/settings.json`); tasks `run tests`/`build tests`/`coverage report`/`presubmit`. **Tests are added one header at a time** (jdf's plan) — infra only so far, with a smoke test.
**Presubmit / push gate**: `./presubmit.sh` (repo root) builds everything + runs the suite, nonzero on failure. **`jj push-main` runs it and aborts the push if it fails** — jj runs NO git hooks, so the gate lives in jdf's global alias (`~/.config/jj/config.toml`), which runs `$(jj root)/presubmit.sh` when it exists and is executable (repos without one are unaffected; a fresh clone is gated with no setup since the alias is global and the script is tracked). `SKIP_PRESUBMIT=1 jj push-main` bypasses. It tests the working copy, not the pushed revision — identical when the working copy is clean. (Still: jdf pushes, Claude never does.)
Coverage = LLVM source-based (Apple clang, via `xcrun llvm-profdata`/`llvm-cov`): `cmake --preset coverage && cmake --build --preset coverage --target coverage` builds an instrumented Debug tree in `build-coverage/` and prints a per-file report scoped to `source/` (+ HTML at `build-coverage/coverage/html/`). ## Device reverse-engineering (the big ongoing thread)
NOT MIDI — it's Roland DT1 SysEx wrapped in a proprietary transport frame over USB CDC-ACM serial (230400 baud). See `[[project-spdsx-patchedit]]` memory for the full protocol map. Cracked so far: kit select, object focus, pad-link, kit name (read+write), pad wave assignment (read+write, nibble-encoded), per-pad layer params (read+write — the kit-record layout above), the full bulk-read handshake, the flash-commit (WRITE) handshake, and the firmware-version query. Device state = ~8MB bulk block transfer on the `41 6c` family, banked; the cached image holds TWO passes of every bank (the app's load runs twice), so per-pass sizes are half the totals. 0x10 = kit/settings (~0.7MB/pass, 3528-byte-per-kit records); 0x20 = the **sample-pool DIRECTORY, not audio** (~3.3MB/pass: 20001 × 164-byte records — wavename/filename/frames/category + an opaque u64 pointer into device flash — then category + user-tag name tables; layout in `device/sample_image.h`). **No wave audio crosses in the state dump at all**, and the official app bundle ships none either: sample download/upload is a separate, still-uncaptured protocol. Method to map a parameter: dump image, change ONE thing in the official app, dump again, diff. Capture + RE tooling lives in **`re/`** (frida capture scripts, `parse_capture.py`, and the original `spdsx.py`/`link_all_kits.py` Python reference this C++ was ported from; see `re/README.md`). frida at `~/frida-env/bin/frida`, SIP disabled; keep attached via `tail -f /dev/null | frida -f <app> -l <script>`. An offline ~8MB device image is cached at **`../re-cache/device-image.bin`** with a memo — the source for all documented storage offsets. jdf drives the GUI + hardware for captures. (Also mirrored at github.com/jdf/spdsx-py.)

**Live transport VERIFIED** (2026-07-11, re-verified after the spdutil rename): `spdutil ping` opened the port and pinged the real device, which replied byte-identically to the frida capture (`f0 41 6a 02 ... 16 40 00 00 00 04 01 00 00 00 f7`). So the whole C++ transport stack (serial open + baud ioctl, frame wrap/unwrap, round-trip) works against hardware. The `/dev/cu.usbmodem<N>` node number changes every replug — there is no stable path; with no `--port`, spdutil scans `/dev/cu.usbmodem*` and pings each node until the device answers. Close the official app first (one program per port). Bulk reads are exercised live too (`spdutil dump` / `kits` / `kit <N>` all ran against hardware 2026-07-12); the write paths (SetKitName/SetPadWave) are still unexercised live.

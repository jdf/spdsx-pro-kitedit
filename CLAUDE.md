# spdsx-patchedit — orientation for Claude

Interactive kit designer for the Roland SPD-SX PRO. C++20, JUCE 8, CMake +
vcpkg. Public repo: github.com/jdf/spdsx-pro-kitedit (note: repo name differs
from this directory). Push with jdf's `jj push-main` alias.

## Conventions (STRICT — jdf cares)
- **Google C++ style throughout**: `.cc`/`.h` files, `#ifndef SPDSX_PATCHEDIT_SOURCE_<FILE>_H_` guards (never `#pragma once`), UpperCamelCase methods (trivial accessors like `name()`/`set_name()` may be snake_case), `kUpperCamelCase` enumerators/constants, `snake_case_` data members. JUCE overrides keep their inherited names.
- **jj (not git) for VCS**, colocated. Commit liberally in small semantic chunks (one logical change per commit, each builds and works; `jj commit <paths>` for per-file chunks). **Ask before pushing** — advance `main` manually then `jj push-main` only with jdf's OK. Co-author trailer: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
- Also honored in sibling repos: `github.com/jdf/specgram` (the spectrogram lib, dir `../specgram`) and `github.com/jdf/spdsx-py` (`../spdsx-py`, the protocol RE).

## Build / run
```
cmake --preset default && cmake --build --preset default
./build/spdsx-patchedit_artefacts/RelWithDebInfo/spdsx-patchedit.app/Contents/MacOS/spdsx-patchedit
```
Targets: the GUI app; `spdutil` (device CLI: `ping` / `info` / `padlink` / `selftest`; `--dry-run` under padlink); `spdsx_device` (JUCE-free static lib). Preset is RelWithDebInfo (breakpoints). VS Code: open `../hax.code-workspace` (multi-root); `--load <slot> <wav>` pre-fills slots for testing.

## Architecture (source/)
- `kit_model.{h}` — the **active kit**: name + 9 `Pad`s, each a top/bottom sample pair plus a `PadParams` (layer mode, fade point/end, dynamics on/off, dynamics curve, trigger reserve — the last is read/write only, not emulated). `PadParams` mutates as a unit: one setter, one action, one listener event. Observable (`KitModel::Listener`). UI/engine/document all react to it.
- `device_model.h` — the **whole-device mirror**: 200 `KitData` (name + pads) as plain data, plus which kit is active. Not observable; the active kit is edited through KitModel and stashed back by the document.
- `layers.{h,cc}` — JUCE-free pure logic emulating the device's 8 layer types (MIX/FADE1/FADE2/XFADE/SWITCH/SW(MONO)/ALTERNATE/HI-HAT): `ComputeLayerWeights(...)` -> per-layer selection weights (0..1) + choke flag, deliberately excluding loudness; `DynamicsGain(curve, velocity)` -> loudness through the Dynamics Curve (LINEAR/LOUD1/LOUD2/LOUD3 — power-curve approximations, real device shapes unpublished). Selection always follows real strike velocity; loudness is weights × DynamicsGain (or × 1.0 with Dynamics off = fixed full volume). Mode/curve names are shared by the UI and `.kit` files.
- `actions.h` — `SetSampleAction`, `SetPadParamsAction` (undoable). `main_component` wraps edits in `undo_` transactions (Load, MoveSample, MovePad, pad-param changes).
- `device_document.{h,cc}` — `DeviceDocument : FileBasedDocument`. The document is a FOLDER `*.spdsx` (a macOS package: Info.plist via CMake `PLIST_TO_MERGE` declares `com.jdf.spdsx-device` conforming to `com.apple.package` with `LSTypeIsPackage`) holding `device.json` (`DeviceFormat` v1: format, currentKit, 200 kits; pads in the old .kit v4 shape; absolute sample paths, pool indices come with device sync; a future `samples/` cache is preserved on save). `SwitchKit` stashes the active kit, loads the next, and preserves the dirty flag (switching is view state). Undo histories are PER-KIT (`MainComponent::undo()` returns the current kit's lazily-created UndoManager) and survive switches — safe because actions only ever enter the manager of the kit that was live, so they replay against the right kit; document-level events (open/new/import) clear all histories via `on_history_reset`. `ImportKitFile` reads legacy single-kit `.kit` v1-v4 into the active kit (File > Import Kit…). ALL device opening goes through `OpenDevice(file-or-folder)` — JUCE's `FileBasedDocument::loadFrom` insists on `existsAsFile()`, which a folder document never satisfies, so File > Open uses its own FileChooser and the app auto-opens the last document on startup (`MainComponent::OpenLastDocument`, skipped when `--load` args or `SPDSX_SELFTEST` are given). **NO EXPLICIT SAVE — everything autosaves** (jdf decision 2026-07-12): every mutation calls `MarkEdited()` (stamps time + `changed()`), the 30Hz timer flushes via `DeviceDocument::Autosave()` after a 1s quiet period, quit flushes synchronously. The document is always file-backed: startup falls back to opening/creating `~/Documents/SPD-SX PRO.spdsx`; File > New asks for a location up front (`CreateNew` saves immediately); no Save menu/cmd-S, no dirty dot/"Edited" marker, no save prompts; Save As remains as "move the document, autosaves follow". `OpenDevice` also accepts the `device.json` path directly (fallback when package registration hasn't kicked in — run `lsregister -f <app>` after building to a fresh location). NOTE: FileBasedDocument's sync save/load are compiled out in JUCE 8 (no modal loops) — selftests go through temporary hooks calling the protected saveDocument/loadDocument.
- `kit_chooser.{h,cc}` — the header's unified kit control: prev/next arrows around a chip with kit number + name; clicking the chip drops the 200-kit menu, the pencil starts in-place rename. Pure view: callbacks out (`on_select`/`on_rename`), `SetCurrent` in, menu names via the `kit_name` provider (live model name for the active kit).
- `main_component.{h,cc}` — the grid, the KitChooser wiring (switching kits via DeviceDocument::SwitchKit), undo, File/Edit/View menus (via ApplicationCommandManager; `main.cc` must call `setApplicationCommandManagerToWatch` or menu enablement goes stale), MIDI in (ch 10 notes 60-68 -> velocity-aware `TriggerPad` through the layer mode; CC4 = hi-hat pedal), click/space anywhere in a pad = whole-pad hit through the layer mode with cursor HEIGHT in the pad as velocity (bottom=1, top=127; slot clicks route via on_click, pad-surface clicks via mouseDown), hits from any source flash the pad in a velocity colour (blue→amber→red, `VelocityColour` in sample_slot) and tint each sounding layer with its adjusted velocity while it plays, keys 1-9 trigger pads at the header VEL slider's velocity, H key held = hi-hat pedal down (press = foot-close "chick": cuts the open layer, sounds the closed one; release silent; key-up detected via `keyStateChanged` + `isKeyCurrentlyDown` polling — `keyPressed` never reports releases), per-pad mode combo + fade sliders + "⋯" menu (Dynamics/Curve/Trigger Reserve) in each pad header, drag (move/copy one slot; ⌘ = whole pad; ⌥ = duplicate). Slot-level transport/space/click stays full-volume, ignoring layer mode.
- `sample_slot.{h,cc}` — a slot: spectrogram + info bar + transport buttons; drag source + FileDragAndDropTarget + DragAndDropTarget; hover highlight driven by parent (`on_drag_target`/`SetDragTarget`).
- `sample_browser.{h,cc}` — left panel FileTree; drags onto slots; autoplay (View menu, persisted). Root folder AND tree state (open folders/scroll/selection, via TreeView openness XML in settings) persist across runs; restore re-applies on a timer until the async directory scans catch up (one shot can't reopen nested folders).
- `audio.{h,cc}` — AudioEngine (per-slot transports + a preview channel), miniaudio-free (pure JUCE).
- `spectro.{h,cc}` — renders spectrogram thumbnails via specgram (consumed through the vcpkg overlay port in `ports/specgram`, pinned to a `github.com/jdf/specgram` sha; bump REF there after pushing specgram).
- `device/` — SPD-SX PRO serial protocol (JUCE-free): `protocol` (DT1 message building, addresses, nibble codec, bulk-read requests + `SplitBulkImage`), `serial_port` (termios + macOS IOSSIOSPEED), `spdsx_device` (frame wrap/unwrap + ops + `DumpBank`), `kit_image` (`CleanBulkImage` strips block framing; `ParseKits` reads the kit-record array). See `spdutil selftest`.
- **Kit record layout** (RE'd 2026-07-12, ground-truth verified: ZZZ kit = #129 and edited pad values read back exactly): bank 0x10, clean (header-stripped) image, array base **0x262c**, stride **3528** bytes, **200** records; name = first **16** bytes (space-padded ASCII). Per-pad param table at **rec+0x284**, 9 pads × **28-byte** blocks; within a block the byte offset = the DT1 param index: +0x00 layer mode (MIX=0…HI-HAT=7), +0x01 fade point, +0x02 fade end, +0x03 dynamics on/off, +0x04 dynamics curve (LINEAR=0…LOUD3=3), +0x05 **fixed velocity** (used when dynamics off — a param the GUI model doesn't have yet), +0x13 trigger reserve. `device/kit_image` parses names + these into `KitRecord`. Waves/EQ/effects/etc. still unmapped (same diff method). `spdutil kits [--from <dump>]` lists names; `spdutil kit <N> [--from <dump>]` shows a kit's pad params.
- **Pad-param WRITE map** (captured live via `re/frida-scripts/paramlog.js`, 2026-07-12): DT1 `06 00 <0x1F+pad> <param>` with the SAME param index as storage, preceded by an object focus `28 00 00 00 <pad-1>`. Values match the enums 1:1. Not yet ported to `spdsx_device` write ops (the "my copy wins" side of sync).
- **No serial commit command** (verified 2026-07-12 via `re/frida-scripts/writecmd.js`, which logs every frame family): the app's WRITE button sends NOTHING on the wire. The DT1 param writes themselves persist — a `06 00 20 01` fade-point edit read straight back in a fresh dump (100→99). So the sync write path is just the DT1 writes; there's no commit to replicate. Caveat: whether an edit survives a device power-cycle without the on-unit WRITE (a device-internal, non-serial flash) is untested.

## Verification pattern (used constantly)
No test target. To verify runtime logic: add a temporary `if (std::getenv("SPDSX_SELFTEST")) { juce::Timer::callAfterDelay(...) { ...checks with PASS/FAIL printf...; std::fflush(stdout); std::_Exit(0); } }` block in the MainComponent ctor, run with `SPDSX_SELFTEST=../specgram/test.wav <app> 2>/dev/null`, then REMOVE before commit. (`_Exit` avoids the quit save-dialog hang; stdout must be flushed.) For device protocol, `spdutil selftest` checks message bytes against captures.

## Device reverse-engineering (the big ongoing thread)
NOT MIDI — it's Roland DT1 SysEx wrapped in a proprietary transport frame over USB CDC-ACM serial (230400 baud). See `[[project-spdsx-patchedit]]` memory for the full protocol map. Cracked so far: kit select, object focus, pad-link, kit name (read+write), pad wave assignment (read+write, nibble-encoded). Device state = ~8MB bulk block transfer on the `41 6c` family, banked (0x10 = kit/settings ~1.4MB with 3528-byte-per-kit records; 0x20 = samples ~6.5MB). Method to map a parameter: dump image, change ONE thing in the official app, dump again, diff. Capture + RE tooling lives in **`re/`** (frida capture scripts, `parse_capture.py`, and the original `spdsx.py`/`link_all_kits.py` Python reference this C++ was ported from; see `re/README.md`). frida at `~/frida-env/bin/frida`, SIP disabled; keep attached via `tail -f /dev/null | frida -f <app> -l <script>`. An offline ~8MB device image is cached at **`../re-cache/device-image.bin`** with a memo — the source for all documented storage offsets. jdf drives the GUI + hardware for captures. (Also mirrored at github.com/jdf/spdsx-py.)

**Live transport VERIFIED** (2026-07-11, re-verified after the spdutil rename): `spdutil ping` opened the port and pinged the real device, which replied byte-identically to the frida capture (`f0 41 6a 02 ... 16 40 00 00 00 04 01 00 00 00 f7`). So the whole C++ transport stack (serial open + baud ioctl, frame wrap/unwrap, round-trip) works against hardware. The `/dev/cu.usbmodem<N>` node number changes every replug — there is no stable path; with no `--port`, spdutil scans `/dev/cu.usbmodem*` and pings each node until the device answers. Close the official app first (one program per port). Read paths (bulk `6c 03` block reads) and write paths (SetKitName/SetPadWave) are still unexercised live — probe/ping is read-only.
